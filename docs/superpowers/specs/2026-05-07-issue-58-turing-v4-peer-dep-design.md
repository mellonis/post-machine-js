# Widen `@turing-machine-js/machine` peer dep to v4 (#58)

**Date:** 2026-05-07
**Issue:** [post-machine-js#58](https://github.com/mellonis/post-machine-js/issues/58)
**Target version:** `@post-machine-js/machine` 4.0.0

## Background

`@turing-machine-js/machine` v4.0.0 shipped on 2026-05-07 (turing-machine-js#100). Its headline change is `TuringMachine.run()` going from synchronous (`void`) to asynchronous (`Promise<void>`). Additive v4 features: `state.debug` per-state runtime-mutable breakpoints, an `onDebugBreak` callback parameter on `run()`, and `haltState.debug` halt-pause.

`@post-machine-js/machine` is currently pinned at peer `^3.0.1`. Issue #58 asks for v4 compatibility.

## Why this is more than a peer-dep widen

The issue's verification step claims "if none of the package's own code calls `run()` or constructs `MachineState`, the v4 break doesn't surface here." That premise is wrong:

- `packages/machine/src/classes/PostMachine.ts:73` overrides `run()` with a `void` return type — incompatible with v4's `Promise<void>` parent signature.
- `packages/machine/src/classes/PostMachine.ts:77` overrides `runStepByStep()`. (This one is fine — v4 keeps `runStepByStep` as a sync `Generator<MachineState>`.)
- `MachineState` is imported and used in callback typings (`packages/machine/src/classes/PostMachine.ts:3`). The shape change in v4 is additive (new optional `debugBreak` field), so no break here.

Therefore widening the peer dep forces `PostMachine.run()` to become async, which is itself a breaking change for `@post-machine-js/machine` consumers — sync `pm.run()` callers must adopt `await`.

## Decisions

- **Peer-dep range:** `"@turing-machine-js/machine": "^4.0.0"`. v3 support dropped. Justification: a major bump on this side is required regardless (see above), and the package will not consume any v4-only feature internally beyond what the override needs, so multi-version support adds testing surface for no gain.
- **Version bump:** `@post-machine-js/machine` → `4.0.0`. Lockstep with the engine major.
- **v4 feature surface:** expose `onDebugBreak` as `__onDebugBreak` on `PostMachine.run()`. The `__` prefix flags it experimental — semantics may change once the per-instruction breakpoint API lands. No other v4 features are wired in at the PostMachine level.
- **Out of scope:** a PostMachine-level per-instruction breakpoint API (e.g. `pm.setBreakpoint(instructionIndex, filter)`). Tracked separately — a new GH issue is filed before this work merges.

## API change

```ts
// post 3.1.0 (against turing 3.x)
override run({ stepsLimit, onStep }: {
  stepsLimit?: number;
  onStep?: (s: MachineState) => void;
} = {}): void

// post 4.0.0 (against turing 4.x)
override async run({
  stepsLimit = 1e5,
  onStep,
  __onDebugBreak,
}: {
  stepsLimit?: number;
  onStep?: (s: MachineState) => void;
  __onDebugBreak?: (s: MachineState) => void | Promise<void>;
} = {}): Promise<void> {
  await super.run({
    initialState: this.#initialState,
    stepsLimit,
    onStep,
    onDebugBreak: __onDebugBreak,
  });
}
```

`runStepByStep` is unchanged.

## Re-exports

`packages/machine/src/index.ts` adds:

```ts
export type { MachineState } from '@turing-machine-js/machine';
```

Rationale: `MachineState` is the parameter type of both `onStep` and `__onDebugBreak`. Without re-export, consumers must take a direct dep on `@turing-machine-js/machine` purely to annotate their callback. That dep is already available via the peer relationship; the re-export is a usability-only addition.

No new value re-exports. `state.debug` mutation, `haltState.debug`, and direct calls to upstream utilities remain available through the peer-dep contract on the consumer side, operating against `pm.initialState`.

## Out of scope (deferred)

A new GH issue is filed before this PR merges, covering: a PostMachine-level API to attach `state.debug` to specific instructions or subroutines (working name `pm.setBreakpoint(instructionIndex, filter)`), with design questions around naming, scoping (per-instance vs per-state-graph), and how it interacts with `state.debug.before` / `.after` filter semantics. The `__onDebugBreak` callback's `__` prefix is the contract that lets that future API rename or restructure the surface without another major bump.
